# AGENTS

本文件用于给后续协作者或智能体说明这个仓库的工作目标、边界和优先级。

## 项目意图

`SmallestTools` 应当逐步演进为一个可复用的 C++ 基础工具仓库，服务于更大的上层项目，例如：

- 模型查看器
- 渲染模拟器
- 资源处理工具
- 文件与日志基础设施

核心目标不是“把每个模块做成独立小程序”，而是“把每个模块做成能被其他工程链接和复用的库”。

## 产物目标

后续工作应围绕下面两个交付目标展开：

- 每个成熟模块都应支持编译为静态库。
- 每个成熟模块都应支持编译为动态库。

如果需要示例程序、调试程序或测试程序，应把它们作为库之外的附加目标，而不是直接把模块本体定义为可执行文件。

## 当前仓库结构

- `FileTools/`
  - 当前最接近“库模块”的目录，已有 `public/private` 分层，但实现仍需加固。
- `ImageTools/PngTool/`
  - 当前是基于 `FileTools` 的 PNG 原型，适合作为图像工具子模块雏形。
- `LogTools/KLogger/`
  - 当前还是占位状态，后续需要先定义清晰 API，再补实现。
- `cmake-build-debug/`
  - 本地 IDE 生成目录，不应作为源码编辑目标。
- `.idea/`
  - IDE 元数据目录，不应作为功能实现的一部分。

## 工作约束

- 不要把模块本体继续扩展成 `add_executable(...)` 风格的主目标。
- 新增功能时，优先把代码放进库模块，而不是写进 `main.cpp`。
- `Sources/public` 只放稳定、对外暴露的头文件。
- `Sources/private` 只放实现细节，不要让外部模块直接依赖私有实现。
- 示例、测试、验证程序应与库目标分离。
- 不要编辑 `cmake-build-debug/` 或 `.idea/` 中的生成文件。

## CMake 约定

后续应优先使用 target-based CMake 写法：

- 使用 `target_sources(...)` 明确列出源码。
- 使用 `target_include_directories(...)` 区分 `PUBLIC` / `PRIVATE` 头文件路径。
- 使用 `target_link_libraries(...)` 描述模块依赖。
- 避免继续使用 `include_directories(...)` 和 `aux_source_directory(...)` 作为主要组织方式。

当库化开始推进时，优先考虑以下结构之一：

- 单一源码列表，同时生成 `static` 与 `shared` 两个目标。
- 共享 `OBJECT` 库，再由静态库和动态库复用同一套对象文件。

## API 与实现约定

- 优先使用 RAII，减少手写原始指针所有权管理。
- 避免硬编码绝对路径。
- 错误处理要么明确返回错误码，要么明确使用异常，不要混用到语义不清。
- 如果准备支持动态库，Windows 下要提前设计导出宏。
- 对外 API 命名应保持统一，避免示例命名和正式模块命名混杂。

## 当前已知薄弱点

- 当前各模块目标仍是可执行文件，不符合仓库目标。
- `FileOperation` 的资源管理和移动语义存在明显风险。
- `PngTool` 还只是早期解析原型，不应视为可稳定复用组件。
- `KLogger` 还没有实际实现。
- 当前本地构建目录存在历史产物，但不能视为稳定发布构建。

## 协作优先级

如果后续要继续建设这个仓库，建议优先顺序如下：

1. 把模块目标从可执行文件调整为库目标。
2. 补齐最小示例与测试，而不是继续扩展空 `main.cpp`。
3. 修复资源管理、移动语义和错误处理问题。
4. 增加安装导出、命名空间和静态库/动态库双产物支持。
5. 再继续扩展 PNG、日志等上层能力。
